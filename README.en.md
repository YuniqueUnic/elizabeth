previews, and share links in one flow.

## Core Capabilities

- **Room-centric collaboration**: Messages, files, and real-time interactions
  all live inside rooms, keeping sharing flows direct and easy to reason about.
- **Single-container, single-port delivery**: One Rust service exposes the API,
  WebSocket endpoint, and embedded frontend assets with a straightforward
  deployment path.
- **Real-time messaging plus rich previews**: Markdown, code highlighting, image
  preview, PDF viewing, text inspection, and link preview are available out of
  the box.
- **SQLite by default, PostgreSQL when needed**: Start with a lightweight local
  database and switch to PostgreSQL through `DATABASE_URL` when scaling up.
- **OpenAPI / Scalar included**: Interactive API docs are available immediately
  for integrations, debugging, and automation.
- **Security baseline included**: JWT auth, room-level access control, and
  least-privilege container settings are built in.

## Quick Start (Docker Deployment, SQLite by default)

Build and deploy with the default, lightweight SQLite database engine:

```bash
# Clone the repository
git clone https://github.com/YuniqueUnic/elizabeth.git
cd elizabeth

# Prepare environment variables file
cp .env.docker .env

# Set a secure JWT_SECRET in production (recommended >= 32 characters)
${EDITOR:-nano} .env

# Start services using Docker Compose
docker compose up -d --build
```

### Access Ports

Once deployed, you can access various entry points:

- **Web UI Interface**: `http://localhost:4092`
- **Scalar Interactive API Docs**: `http://localhost:4092/api/v1/scalar`
- **OpenAPI Specs (JSON)**: `http://localhost:4092/api/v1/openapi.json`
- **Health Check**: `http://localhost:4092/api/v1/health`

## Advanced Database Configuration (Adaptive Driver Switch)

Leveraging `sqlx::AnyPool`, Elizabeth automatically decides which engine to use
and applies database migrations dynamically based on the schema of
`DATABASE_URL`:

- **SQLite** (Default, reads `./crates/board/migrations` / `/app/migrations` in
  container)
- **PostgreSQL** (Reads `./crates/board/migrations_pg` / `/app/migrations_pg` in
  container)

### Deploy with Pre-configured PostgreSQL Container

To spin up Elizabeth along with a PostgreSQL container in the same environment:

```bash
docker compose -f docker-compose.yml -f docker-compose.postgres.yml up -d --build
```

### Connect to an External PostgreSQL

If you prefer to connect to an external PostgreSQL server, update `.env` to
point to it (ensure network accessibility):

```env
DATABASE_URL=postgresql://username:password@hostname:port/dbname # pragma: allowlist secret
```

## Key Configurations

In a Docker containerized environment, use the `.env` file at the root to
configure crucial settings:

| Environment Variable | Default Value                     | Description                                                               |
| :------------------- | :-------------------------------- | :------------------------------------------------------------------------ |
| `JWT_SECRET`         | (Required)                        | Key used to sign authentication tokens. Always change this in production. |
| `DATABASE_URL`       | `sqlite:///app/data/elizabeth.db` | DB connection string. The schema determines the driver selection.         |
| `DB_MAX_CONNECTIONS` | `20`                              | Maximum number of active db connections in the pool.                      |
| `DB_MIN_CONNECTIONS` | `5`                               | Minimum number of idle connections to retain in the pool.                 |

> [!NOTE]
> The static configuration template resides in
> `docker/backend/config/backend.yaml` and is loaded during runtime (YAML files
> do not support dynamic environment variable interpolation; secure variables
> must be configured via environment overrides).
